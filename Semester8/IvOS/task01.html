Zadání

Stáhněte si základní funkční implementaci plánovače Round-Robin (RR) z http://ivos.mrl.cz/download/gthreads.tgz (Stahujte z VPN pokud jste mimo TUO-NET)

Archiv obsahuje Makefile a po rozbalení byste měli být schopni přiložený projekt přeložit a spustit. Následuje ukázkový výpis se stažení, překladem a spuštěním v Linuxu:

Překlad

[kru13@localhost ivos]$ make
cc -g -Wall   -c -o gthr.o gthr.c
as -o gthr_switch.o gthr_switch.S
cc -g -Wall   -c -o main.o main.c
cc -o gthr_demo gthr.o gthr_switch.o main.o

Spuštění

[kru13@localhost ivos]$ ./gttest  
F Thread id = 1, val = 1 BEGINNING
F Thread id = 2, val = 1 BEGINNING
G Thread id = 1, val = 1 BEGINNING
G Thread id = 2, val = 1 BEGINNING
G Thread id = 1, val = 2 END
F Thread id = 1, val = 2 END
F Thread id = 2, val = 2 END
G Thread id = 2, val = 2 END
F Thread id = 1, val = 3 BEGINNING
F Thread id = 2, val = 3 BEGINNING
G Thread id = 1, val = 3 BEGINNING
G Thread id = 2, val = 3 BEGINNING
^C

Projděme si nyní komentované zdrojové soubory. V souboru main.c se nalézají dvě ukázkové funkce f() a g(), které nám poslouží jako těla našich testovacích vláken. Obsahují testovací nekonečné smyčky s výpisy indikujícími o jaké vlákno a kolikátou iteraci běhu se jedná. V hlavní metodě main() máme inicializaci našeho plánovače skrz gt_init() a následné vytvoření čtyř vláken pomocí metody gt_create(), které se jako parametr předává funkce, která má být spuštěna jako tělo vlákna (analogie k metodě run v Javě). Poslední volání gt_return() nám zajišťuje, že se main() okamžitě neukončí, ale bude čekat až na doběhnutí všech aktivních vláken.

V hlavičkovém souboru gthr.h, naleznete strukturu gt popisující stav (kontext) každého vlákna. Ve struktuře jsou aktuálně uloženy hodnoty registrů procesoru a stav vlákna (Unused, Running, Ready). Tato struktura pak tvoří jednotlivé položky v tabulce vláken (obdoba process table a process control block, jak jej znáte z přednášky o procesech). Tabulka gt_table[] (v soubor gthr_struct.h) je staticky alokována a hodnota MaxGThreads definuje jaký je maximální počet podporovaných vláken.

V gthr.c najdeme veškerou implementaci našeho jednoduchého plánovače a pomocných funkcí pro manipulaci s vlákny. V metodě gt_init() najdeme inicializaci prvního vlákna a nastavení alarmu, signálu, který nám pravidelně bude přerušovat tok zpracovávání vláken a umožní nám přepínat mezi vlákny. Při vytváření nového vlákna se volá metoda gt_create() která najde postupným procházením v tabulce gt_table[] první volné místo pro nově vytvářené vlákno, alokuje paměť pro stack tohoto vlákna a inicializuje návratové adresy na vrchol takto vytvořeného stacku (tím zajistíme, že při zavolání return v rámci vlákna se dostaneme do řízeného ukončení skrz metodu gt_stop()). Klíčovou metodou našeho plánovače je metoda, která rozhoduje, jaké vlákno poběží jako další po přepnutí kontextu. Jedná se o gt_schedule(), která implementuje jednoduchý plánovač typu Round-Robin. V kódu nalezneme kruhové procházení tabulky gt_table[], kde hledáme další vlákno, které je schopno běhu (je ve stavu Ready) a poté samotné přepnutí kontextu, které je jednak realizováno aktualizací hodnot v tabulce gt_table[] tak i samotné přepnutí přepsáním hodnot registru procesoru (funkce gt_switch(), která je jako jediná implementována v assembleru v souboru gt_switch.S).

Postup

Prvním úkolem je seznámit se s tímto projektem, vyzkoušet si, jak se chová a poté modifikovat stávající kód tak, aby reagoval na stisk kláves CTRL+C (SIGINT) a vypsal tabulku se statistikou běžících vláken.

  • Do struktury gt (gthr.h) přidejte nové položky, které Vám umožní sledovat dobu běhu daného vlákna, dobu čekání na procesor, minimální, maximální a průměrnou hodnotu a jejich rozptyl. Pro měření času je vhodné použít např. gettimeofday
  • Dodejte obsluhu signálu SIGINT a odpovídající výpis statistik jednotlivých vláken.

Příprava plánovače pro integraci do vlastního operačního systému

Cílem této úlohy není pouze vytvořit funkční implementaci plánovače vláken v prostředí Linuxu, ale především navrhnout takové řešení, které bude možné později přímo integrovat do vašeho vlastního operačního systému. Při implementaci proto berte v úvahu následující požadavky a omezení cílového prostředí.

Nezávislost na operačním systému a knihovnách

Vaše výsledná implementace musí být přenositelná do prostředí bez standardní knihovny jazyka C (freestanding). To znamená, že:

  • nelze spoléhat na funkce jako malloc, free, printf, signal, alarm apod.,
  • veškeré I/O operace musí být realizovatelné přímo (např. zápisem do paměti nebo přes porty),
  • kód musí být kompilovatelný jako „freestanding binary“.

Doporučení: Oddělte části závislé na platformě (např. časovač, výpisy, správa paměti) do samostatné vrstvy.

Správa zásobníků (stacků)

V cílovém OS nebude dostupný dynamický alokátor paměti (v aktuální fázi vývoje). Proto:

  • nepoužívejte malloc pro alokaci stacku,
  • implementujte statickou alokaci zásobníků, např.: 
    #define STACK_SIZE 4096
uint8_t stacks[MAX_THREADS][STACK_SIZE];

Ukončení vláken

Vlákna nesmí končit „nekontrolovaně“ návratem z funkce. Zajistěte:

  • řízené ukončení pomocí gt_exit(),
  • uvolnění slotu v tabulce vláken,
  • možnost opětovného využití vlákna.

Rozhraní plánovače (API)

Navrhněte rozhraní tak, aby bylo použitelné i v kernelu. Doporučené funkce:

void gt_init();
int  gt_create(void (*entry)(void*), void* arg);
void gt_yield();
void gt_exit(int code);
void gt_schedule();

Pro budoucí integraci dále připravte: 

void scheduler_init();
void scheduler_tick();   // voláno z časovače (IRQ)
